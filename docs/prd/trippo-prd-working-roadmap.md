## 1. 产品概述

### 1.1 产品是什么

Trippo 是一个围绕“一次旅行”组织体验的移动端产品，核心价值是把旅行前的收集、旅行中的执行记录、旅行后的沉淀回看串成闭环：

**心愿单 → 行程 → 路线与打卡 → 旅行手账**

它不是机酒预订平台，不是重社交社区，也不是泛日记工具；它更像一个“旅行陪伴”与“旅行记忆管理”的产品。

### 1.2 一句话定位

**懂你的旅行陪伴助手**：让一次旅行从「想去哪」到「回忆沉淀」都被完整保存下来。

## 2. 当前边界（非常重要）

### 2.1 已实现（代码仓库真实存在）

当前仓库是“先把闭环跑通”的实现形态：

- 无后端、无账号：纯前端 + `localStorage` 本地持久化
- 无地图 API：景点搜索基于 mock 数据池
- 已实现模块（V1.0 + V2.0）：
  - 行程（列表/创建/详情）
  - 心愿清单（行程内）
  - 每日路线（按天/排序/从心愿单加入）
  - 打卡（编辑/列表）
  - 手账（生成草稿/编辑/列表/详情）
  - 主题清单（List/清单列表+详情+进度+转行程）
  - 花费分类与统计（按分类/按天/总计）
  - 手账模板（经典/旅行/可爱/极简）

### 2.2 蓝图规划（PRD 统一版里写了，但当前没落地）

这些是长期设想，但当前仓库并未实现：

- 登录/云同步/多端
- 高德 POI 数据接入

- 独旅模式（安全/节奏/独拍）
- 协作、AI 辅助、导出分享、智能排序

## 3. 用户与场景

### 3.1 核心用户

- 一年有 2 次以上自由行的普通旅行者
- 爱拍照、愿意打卡、愿意留一点感受，但不想使用多个工具拼凑
- 旅行后希望低成本整理出一份能回看的“作品”

### 3.2 典型场景（按时间线）

1. 出发前：收集想去的（景点/美食/旅行地/主题清单）
2. 出发前：创建一趟行程（目的地、日期）
3. 出发中：把心愿加入每天路线并排序
4. 出发中：到点打卡（照片/花费/感受）
5. 回来后：基于打卡素材生成手账并整理

## 4. 产品设计（你后面改版主要看这一章）

### 4.1 设计原则

- 像常规 App：核心入口清晰、固定、可预测
- 移动端优先：单手可操作，底部主入口稳定
- 先清楚再精致：先把路线与入口讲清楚，再谈视觉升级
- 结构先于美化：先把信息架构改顺，再做 UI 抛光

### 4.2 信息架构（最新验收结论）

#### 4.2.1 底部主入口

Trippo 的底部主导航应收拢为 3 个（主线一眼看懂）：

- 行程
- 心愿单
- 旅行手账

说明：

- **心愿单**与**旅行手账**都是重要能力，必须一级入口
- **主题清单（List）**不再单独当“底部入口”，而是**内嵌在心愿单**（心愿单的一种组织方式）

#### 4.2.2 双入口规则（全局视图 vs 行程视图）

同一个功能要支持两种进入方式，且两种进入方式看到的内容不同：

1. 全局入口（底部导航进入）
   - 看“全部内容”，按行程分类 + 未关联行程的归类
2. 行程内入口（从某个行程详情进入）
   - 只看“这趟旅行相关内容”

覆盖范围：

- 心愿单：全局心愿单 / 某行程心愿单
- 旅行手账：全局手账 / 某行程手账

#### 4.2.3 行程首页（行程列表页）展示规则

进入行程页后，应按状态分区展示，突出“当前这一趟”：

- 进行中：置顶、视觉最突出
- 计划中：最多展示 2–3 条 + “查看更多”
- 已完成：最多展示 2–3 条 + “查看更多”

行程状态建议统一为：

- 计划中
- 进行中
- 已完成

### 4.3 核心功能结构（目标状态）

#### 4.3.1 行程

用户管理“具体一趟旅行”的入口：

- 行程列表（按状态分区）
- 创建行程
- 行程详情（中枢）
  - 该行程心愿单（局部视图）
  - 每日路线（按天/排序）
  - 打卡（从路线/详情进入）
  - 该行程手账（局部视图）

#### 4.3.2 心愿单

用户管理“我想去/想吃/想体验”的入口：

1. 分类维度（你提出的核心要求）
   - 景点心愿
   - 美食心愿
   - 旅行地心愿
   - 清单（主题清单内嵌）
2. 行程维度
   - 按行程查看
   - 未关联行程（合理归类）

从行程进入心愿单时：只显示该行程相关心愿。

#### 4.3.3 旅行手账

用户回看与沉淀旅行记忆的入口：

- 全部手账（按行程分类）
- 草稿（未完成/未归档）
- 未关联行程内容（合理归类）

从行程进入手账时：只显示该行程手账。

已确认的“必须改”点（来自验收）：

- 主入口必须像常规 App 一样放到底部：行程/心愿单/旅行手账

- 手账必须独立入口，并支持按行程分类
- 心愿单与手账必须支持“双入口”：全局/行程内

## 7. 后续迭代计划（每次迭代怎么做）

下面把后续计划拆成“可连续交付的小版本”。你可以按实际节奏增删、合并。

> 约定：每次迭代都写清楚「目标」「范围」「不做」「验收标准」四件事，后面改起来才不会跑偏。

### V2.1（优先）主入口改版

目标：把 Trippo 改成“像常规 App”，用户一眼看懂三大入口。

范围：

- 底部导航改为：行程 / 心愿单 / 旅行手账
- 行程首页分区展示：进行中、计划中、已完成（计划中/已完成各最多 2–3 个 + 查看更多）
- 行程详情页补充：
  - 进入“该行程心愿单”
  - 进入“该行程手账”

不做：

- 不引入推荐/发现页
- 不做账号/云同步
- 不做大规模 UI 换肤

验收标准（可直接同步到 `tests/acceptance-v2.md`）：

- 底部三入口清晰可用
- 行程首页分区规则符合预期
- 进入行程后能跳到该行程心愿单/手账，并且是“局部视图”

### V2.2 心愿单改版（分类 + 行程维度）

目标：把心愿单做成真正的一级入口。

范围：

- 心愿单入口页支持：
  - 分类：景点心愿 / 美食心愿 / 旅行地心愿 / 清单
  - 按行程查看（含未关联行程归类）
- 清单（List）能力在心愿单内可访问（不再“另一个世界”）
- 行程内进入心愿单：默认只显示该行程相关内容

不做：

- 不做复杂推荐内容库
- 不做跨设备同步

验收标准：

- 从底部进入心愿单能“按类型看”也能“按行程看”
- 未关联行程内容有合理归类
- 从行程进入心愿单只展示局部内容

### V2.3 旅行手账改版（独立入口 + 按行程分类）

目标：让手账成为 Trippo 的“价值完成点”，并具备长期回看的结构。

范围：

- 手账入口页支持：
  - 全部手账（按行程分组）
  - 草稿
  - 未关联行程内容归类
- 行程内进入手账：只显示该行程手账

验收标准：

- 全局入口能按行程分类回看
- 行程入口能只看当前行程
- 草稿与已完成手账表达清楚

### V2.4（可选）体验与视觉抛光

目标：在结构稳定后，统一 UI 细节，提升完成度。

范围：

- 统一按钮/卡片/输入框/空状态/导航样式
- 统一标签（行程状态、心愿类型、打卡分类、模板标签）
- 优化关键路径提示文案

验收标准：

- 核心路径无明显割裂感
- 空状态不再“像没做完”

### V3.0 智能推荐（先心情推荐）

目标：解决“我不知道去哪”的决策入口，把灵感转为心愿或行程。

范围：

- 心情推荐入口
- 推荐结果可解释
- 一键加入心愿单 / 一键转行程

不做：

- 不承诺“算法很准”，定位为趣味偏好推荐

验收标准：

- 用户能从推荐结果顺滑进入“心愿单/行程”

### V4.0 独旅模式

目标：服务独旅人群的安全感与节奏需求（不做陌生人社交）。

范围：

- 独旅设置（紧急联系人/提醒）
- 路线强度提示
- 独拍/独处友好提示（轻量）

### V5.0 协作与 AI 增强

目标：在产品成熟后再做协作与 AI，不让前期复杂度拖累核心体验。

范围：

- 手账共编、清单协作
- AI 辅助生成/润色（需兜底）
- 导出分享与隐私预览
