## 1. 产品概述

### 1.1 背景描述

旅行是一件高频且情绪浓度很高的事，但围绕一次旅行的「**计划 — 出行 — 记录**」三个环节，今天往往被割裂在不同工具里：行程靠备忘录或截图拼凑，景点和路线在地图、攻略、社交平台之间反复横跳，回来后想认真写一篇游记又因为素材散落而难以动笔。用户真正需要的，是一个能把「**一次旅行**」当作完整对象来承载的轻量工具——出行前能快速搭出行程与打卡路线，出行中能顺手打卡记录，回来后能低成本沉淀成一份好看的旅行手账。

### 1.2 当前痛点

- **痛点 1 · 计划分散**：航班、酒店、景点、每日安排散落在备忘录、聊天框、地图收藏与攻略截图里，出发前难以整理成一条可执行路线。
- **痛点 2 · 景点与路线脱节**：攻略里看到的景点是一个个孤立的点，缺少「按天串成顺路打卡路线」的能力，到了当地容易走回头路。
- **痛点 3 · 旅途记录断裂**：旅途中拍的照片、花的钱、当下的感受是分散的，地点、时间、路线与心情没有自动连在一起，事后难以还原。
- **痛点 4 · 沉淀门槛高**：想认真写一篇游记/手账，从零排版、找素材、组织内容成本很高，多数人最终不了了之。
- **痛点 5 · 出行前「不知道去哪」**：大量用户的真实痛点在出发前的选择困难，通用工具默认用户已经知道去哪，决策环节缺位。
- **痛点 6 · 不同人群诉求被忽视**：独旅、情侣、闺蜜、亲子的诉求完全不同，尤其独行者长期被「默认多人出行」的工具忽视。

### 1.3 产品定位

通过「**一次旅行 = 一个完整容器**」的产品设计，把行程、景点路线、打卡记录、手账串成闭环；并向前延伸到「**帮你决定去哪、陪你一起期待**」，向人群差异化延伸到「**懂你身份、关系与心情**」。

<grid>
<column width-ratio="0.500000">
**基础定位（核心闭环）**
一款帮普通旅行者轻量规划行程与打卡路线、便捷记录旅途、低门槛沉淀旅行手账的移动端**记录工具**，价值集中在「出行中 + 出行后」。
</column>
<column width-ratio="0.500000">
**升级定位（特色模块）**
一款懂用户身份、关系与心情，能主动推荐「去哪、和谁、怎么玩」的**旅行陪伴助手**，价值链向「出行前」延伸，覆盖旅行全生命周期。
</column>
</grid>

<callout emoji="⭐">
**一句话定位**：从「记录一次旅行的工具」升级为「**懂你的旅行陪伴助手**」——让一次旅行从「想去哪」到「回忆沉淀」都被完整、好看地保存下来。
</callout>

### 1.4 产品目标与成功指标

**MVP（V1.0）核心目标：**

- **目标 1 · 规划省心**：用户能在 3 分钟内创建一次行程，并完成至少一天的景点选择与打卡路线规划。
- **目标 2 · 记录省事**：用户在旅途中能在 30 秒内完成一次景点打卡（含照片 / 花费 / 感受）。
- **目标 3 · 沉淀好看**：用户旅行结束后能基于已有打卡素材，半小时内产出一份图文旅行手账。

**北极星指标**：**完成旅行记录的旅行数**——即一次旅行中至少包含 1 天行程、3 个景点、2 次打卡和 1 份手账草稿，用于衡量「规划—记录—沉淀」闭环是否真正跑通。

| 指标类型 | 指标名称 | 目标值 / 判断方式 |
|-|-|-|
| 规划效率 | 创建行程并完成首日路线规划耗时 | ≤ 3 分钟 |
| 记录效率 | 单次景点打卡耗时 | ≤ 30 秒 |
| 闭环完成率 | 创建行程后最终产出手账的比例 | ≥ 30%（种子用户阶段） |
| 激活 | 创建行程后 24 小时内添加 ≥ 3 个景点的用户占比 | 内测后确认目标值 |
| 留存意愿 | 同一用户创建第 2 次行程 | 有明确复用 / 二次创建行为 |

## 2. 竞品分析

公开信息显示，市面产品分别覆盖了「计划、轨迹、行程聚合、日记、攻略社区」等环节，但「**从计划到打卡再到手账**」的一体化体验，叠加「**人格/心情推荐 + 主题清单**」的差异化能力，仍存在明显空间。

| 产品 | 公开定位 | 对本产品的启发 | 可避开的方向 |
|-|-|-|-|
| Wanderlog | 旅行计划、行程和地图工具 | 计划与地图需深度结合，路线不是附属功能 | 避免第一版做过重的协作和预订管理 |
| Polarsteps | 旅行规划、轨迹记录、旅程回看 | 旅行记录可自动生成时间线，降低补录成本 | 不要把轨迹做成唯一核心，否则手账创作空间不足 |
| TripIt | 自动组织旅行行程 | 结构化行程对商务与长途旅行很有价值 | 邮箱导入 / 票务解析可后置，MVP 先手动创建 |
| Day One | 多媒体日记与个人数据保护 | 手账编辑需支持照片、文字、隐私设置 | 不要做泛日记，旅行语境必须足够强 |
| 马蜂窝 | 攻略、图片、游记与自由行信息社区 | 游记内容与地点信息可互相补充 | 第一版不做开放社区，避免审核与冷启动压力 |

<callout emoji="💡">
**差异化结论**：本产品不做重交易的旅行平台，也不做单纯的日记工具，而是「**以一次旅行为单位管理个人记忆**」的工具，并通过「心情/人格推荐 + 主题清单 + 独旅关怀」形成有辨识度的产品性格。MVP 阶段坚持克制：**不做机酒预订、不做复杂社交关系、不做达人内容社区**。
</callout>

## 3. 目标用户与用户画像

### 3.1 核心用户

以 **18–35 岁、有自主出行习惯的年轻旅行者**为核心：他们会自己做攻略、规划行程，旅途中爱拍照打卡，回来后有记录意愿，但缺少一个把全流程串起来的趁手工具。更精确的切口是：**一年有 2 次以上自由行、愿意拍照记录、会在旅行前收藏攻略但不一定擅长整理的人**。

### 3.2 用户角色与画像

| 角色 | 典型特征 | 高频场景 | 痛点强度 |
|-|-|-|-|
| 独立旅行者 | 自己规划路线，偏好自由行，常在手机上查攻略和地图 | 出发前收集景点；当天按体力调整路线；晚上整理照片 | 高 |
| 情侣 / 朋友小团体 | 一起出行，路线由 1 人主导，其他人参与选择 | 投票选景点；共享当天路线；旅行后共同补充照片文字 | 中 |
| 轻内容创作者 | 愿意产出图文游记，但不一定追求公开流量 | 旅行后整理为图文手账，导出图片或链接分享 | 中高 |
| 本地周末探索用户 | 短途出行频率高，每次行程较短 | 周末选择 2–4 个点打卡，记录咖啡馆、展览、公园等 | 中 |
| 独旅 / 个性化人群 | 独自出行（尤其女性独行者）、热衷 MBTI / 星座等身份标签 | 担心安全、独自拍照不便；好奇「我这类人适合去哪」 | 中高 |

### 3.3 典型使用场景

- **场景 1 · 出行前规划（Before）**：用户确定目的地与日期，创建行程，浏览并挑选景点加入心愿清单，再按天排成顺路打卡路线。
- **场景 2 · 旅途中打卡（During）**：用户到达景点后快速打卡——拍照、记花费、写感受，系统自动归属到对应行程当天。
- **场景 3 · 旅行后沉淀（After）**：旅行结束后进入手账模块，基于打卡素材套用模板快速排版，生成图文手账并回顾。
- **场景 4 · 灵感决策（Pre-Before）**：用户「不知道去哪」时，从心情/人格推荐获得一组旅行灵感，一键转为心愿清单或行程。
- **场景 5 · 长期愿望管理**：用户在两次旅行之间，通过主题清单（心愿/遗愿/情侣/闺蜜/亲子）持续收集愿望，让产品在「没有旅行时」也有打开理由。
- **场景 6 · 独自旅行**：一个人去陌生城市，担心安全、想拍好看的照片、想要轻松不赶场的节奏与独处友好的推荐。

## 5. 功能详细需求

> 本章是 PRD 核心，按「功能描述 / 关键规则 / 字段设计 / 验收标准」组织。**核心功能（F1–F13）写细，特色功能（F14–F34）适当概要**。

### 5.1 账号与同步（F1–F2）

#### F1 注册 / 登录与云账号体系

- **功能描述**：用户需注册 / 登录后使用，所有行程、景点、打卡、手账、清单等数据均以「用户账号」为顶层归属并云端存储。
- **关键规则**：

  - 支持手机号 / 第三方账号登录（具体方式由开发侧定）；登录态具备安全校验与失效保护，避免被冒用。
  - 提供清晰的登录 / 退出 / 注销入口；首次使用涉及个人信息的功能时进行权限与隐私说明。
  - 数据默认仅本人可见，第一版不做对外公开。
- **验收标准**：用户完成注册 / 登录后，能在任意已登录设备访问到归属自己账号的全部数据；退出后本地不残留敏感明文数据。

#### F2 多端云端同步 + 弱网本地缓存

- **功能描述**：同一账号在多端登录时数据一致同步；本地做缓存以支持弱网 / 断网下关键操作可用。
- **关键规则**：

  - 行程、景点、打卡、手账、清单数据云端为准，多端登录后保持一致同步，避免端间冲突或丢失。
  - 弱网 / 断网下，打卡、手账编辑、清单收藏等关键操作支持本地暂存，恢复网络后自动回传云端。
  - 照片上传支持压缩与异步处理，失败可重试，不丢失内容。
- **验收标准**：断网下完成的打卡 / 编辑，在恢复网络后能正确同步至云端并在其他端可见；冲突场景有明确的合并 / 提示策略。

### 5.2 行程记录与管理（F3–F6）

#### F3 创建 / 编辑行程

- **功能描述**：用户新建一次行程（目的地 + 起止日期），系统按天数自动生成空白行程框架，可编辑标题、封面。
- **关键规则**：

  - 目的地与起止日期为必填；开始日期不晚于结束日期；行程标题留空时默认「目的地 + 月份」自动生成。
  - 创建后默认状态「即将出发」；当前日期落入行程区间内自动变「进行中」；结束日期后若存在 ≥ 1 条打卡或手账草稿，变「已完成」。

**字段设计 · 行程**

| 字段 | 类型 | 是否必填 | 说明 |
|-|-|-|-|
| 目的地 | 文本 / 选择 | 是 | 支持输入城市名，可后续扩展为多目的地 |
| 开始日期 | 日期 | 是 | 不晚于结束日期 |
| 结束日期 | 日期 | 是 | 用于自动计算行程天数（留空则默认单日） |
| 行程标题 | 文本 | 否 | 留空默认「目的地 + 月份」生成；1–40 字符 |
| 封面图 | 图片 | 否 | 留空使用目的地默认封面 |

- **验收标准**：填写目的地与起止日期后可成功创建并自动生成按天框架；必填缺失时就地高亮提示、不产生脏数据；保存失败保留已输入内容并可重试。

#### F4 行程列表与详情

- **功能描述**：以列表统一管理所有行程，区分「即将出发 / 进行中 / 已完成」；进入详情查看每日安排、每日打卡路线、打卡入口与花费汇总。
- **关键规则**：列表按时间排序，卡片展示目的地、日期、完成进度、最近一次记录与封面；无任何行程时展示引导卡片，突出「创建第一次行程」入口。
- **验收标准**：列表能正确分组并按时间排序；详情页能正确展示该行程的天、景点、打卡与花费汇总。

#### F5 景点打卡

- **功能描述**：用户到达景点后打卡，记录照片、花费、感受、时间与可选定位，自动归属到对应行程当天，沉淀为手账素材。
- **关键规则**：

  - 打卡可从当天路线选择景点，或新增临时景点；未授权定位时可手动完成打卡，记录标记「未验证位置」。
  - 状态流转：未安排 → 已计划 → 当天待到达 → 已到达 → 已记录；删除打卡记录后景点回到「当天待到达」，计划顺序不变。
  - 弱网下支持本地暂存与恢复后重试上传。

**字段设计 · 打卡记录**

| 字段 | 类型 | 是否必填 | 说明 |
|-|-|-|-|
| 关联景点 | 选择 / 文本 | 是 | 从当天路线选择，或新增临时景点 |
| 照片 | 图片（多张） | 否 | 支持多张，作为手账素材 |
| 花费 | 数字 | 否 | 记录单笔花费，用于行程花费汇总 |
| 感受 | 长文本 | 否 | 记录当下心情 / 备注 |
| 打卡时间 | 系统生成 | 是 | 默认当前时间，可手动修正 |
| 定位 | 地理坐标 | 否 | 授权后自动获取，未授权标记「未验证位置」 |

- **验收标准**：用户可对景点打卡，保存后正确归属到对应行程当天；定位权限关闭时仍能手动记录；弱网内容可本地暂存并恢复后重试。

#### F6 花费统计与汇总

- **功能描述**：基于打卡花费字段汇总单次行程总花费（V1 基础求和）；V2.0 完善为按天 / 分类统计、人均与可视化。
- **关键规则**：花费来源于打卡记录的花费字段；V2 支持分类标签（餐饮 / 交通 / 门票 / 住宿 / 其他）。
- **验收标准**：行程详情能正确展示总花费；V2 能按维度拆分统计且与打卡数据一致。

### 5.3 景点选择与打卡路线（F7–F11）

#### F7 景点浏览 / 搜索

- **功能描述**：在某次行程下浏览 / 搜索景点（基础数据接入**高德地图开放平台 API** 作为 MVP 数据底座），支持搜索与基础筛选。
- **关键规则**：

  - 景点基础信息（名称、位置、分类等）来自高德 API；**搜索无结果时支持用户自定义添加（名称即可）作为兜底**。
  - 缺少地址的自定义景点允许保存，但需提示「该景点无法参与路线 / 导航计算」。

**字段设计 · 景点**

| 字段 | 类型 | 是否必填 | 说明 |
|-|-|-|-|
| 景点名称 | 文本 | 是 | 来自高德 API，或用户自定义 |
| 所属城市 / 区域 | 文本 | 否 | 便于筛选与归类 |
| 地址 / 坐标 | 文本 / 坐标 | 否 | 高德返回；缺失则不参与路线与导航 |
| 分类标签 | 标签 | 否 | 如自然、人文、美食、网红打卡等 |
| 所属天数 | 选择 | 否 | 规划路线时分配，未分配则留在心愿清单 |
| 排序序号 | 数字 | 否 | 同一天内的打卡先后顺序 |

- **验收标准**：用户可浏览 / 搜索景点；搜索无结果时可手动添加自定义景点并正常加入流程。

#### F8 心愿清单收集

- **功能描述**：景点先进入心愿清单暂存，与具体某天解耦，方便先收集再编排。
- **关键规则**：清单数量正确累加；清单为空时路线规划页展示空状态并引导去选景点。
- **验收标准**：景点加入 / 移除心愿清单数量准确，且可持久保存。

#### F9 打卡路线规划

- **功能描述**：把心愿清单景点按天分配并通过拖拽排序，生成每日打卡路线，以「按天 + 有序列表」呈现。
- **关键规则**：

  - 路线按行程天数分组；同一天内拖拽调整先后顺序，顺序即打卡建议顺序。
  - 同一景点可重复加入不同日期，但需提示「已在 D2 安排过」；从某日移除不删除原始资料，仍保留在候选池。
  - 当天景点超过阈值（如 10 个）显示「行程可能过满」提醒，但允许继续。
- **验收标准**：可把心愿清单景点分配到某一天并拖拽调整顺序，结果可持久保存；空清单时有空状态引导。

#### F10 第三方地图导航跳转

- **功能描述**：**App 内不做内置地图导航**；地点详情支持一键唤起手机内第三方地图 App（**高德 / 百度地图**）进行实际导航。
- **关键规则**：App 内路线仅做按天有序列表呈现，不自建导航能力；无坐标的自定义景点不提供跳转。
- **验收标准**：有坐标的地点可一键跳起高德 / 百度地图并定位到对应地点；无坐标时按钮置灰或提示补充地址。

#### F11 行程内搜索与筛选

- **功能描述**：在某次行程内按景点名、标签、日期、记录内容搜索筛选，结果可跳转到景点详情 / 时间线记录 / 手账段落。
- **验收标准**：搜索结果准确且可正确跳转目标位置。

### 5.4 旅行手账（F12–F14）

#### F12 手账编辑器（个人版）

- **功能描述**：用户选择某次行程进入手账编辑器，可套用模板，系统自动带入该行程打卡素材（照片、足迹、花费、感受），图文混排编辑。
- **关键规则**：

  - **素材引用**：一键插入该行程打卡照片与文字片段，减少重复劳动。
  - **结构**：以「段落块」为单位，支持文字块 / 图片块的增删与排序；编辑器支持自动保存。
  - **多人共编**：第一版定位为个人创作，**不支持多人共编**；后续版本（F29）基于云账号体系迭代支持。

**字段设计 · 手账**

| 字段 | 类型 | 是否必填 | 说明 |
|-|-|-|-|
| 手账标题 | 文本 | 是 | 默认带入行程标题，可修改 |
| 关联行程 | 选择 | 是 | 手账所属的行程（一对多：一次行程可写多篇） |
| 模板 | 选择 | 否 | 默认极简模板，可切换 |
| 内容块 | 图文块列表 | 是 | 文字块 / 图片块组成，可增删排序 |
| 更新时间 | 系统生成 | 是 | 记录最近编辑时间 |

- **验收标准**：可选择行程进入编辑器并成功带入打卡素材；可对内容块增删排序、套用 / 切换模板；退出后自动保存；无打卡素材时也能从空白创建。

#### F13 手账保存与查看

- **功能描述**：手账保存成册并归属到对应行程，可在行程详情或手账列表随时打开回顾。
- **验收标准**：保存后正确归属行程并可再次打开；归属关系正确。

#### F14 手账模板库丰富（V2.0）

- **功能描述**：在 MVP 少量模板基础上扩充模板库（时间轴式、图文卡片式、独旅个人叙事式等），降低排版门槛。
- **验收标准**：模板可切换且切换后内容不丢失。

### 5.5 主题清单体系（F15–F18，V2.0）

> 主题清单是「心愿清单」的升级与扩展：从普通收藏夹升级为**有情感、有主题、自带内容**的清单体系，让产品在两次旅行之间也有打开理由。**第一版（V2.0）为个人版，不支持多人协作（见 F30）。**

**清单类型设计**

| 清单类型 | 定位 | 内置推荐内容示例 |
|-|-|-|
| 心愿清单 | 这辈子想去的地方，长期收集 | 此生必去的 50 个目的地、世界级自然奇观 |
| 遗愿清单（人生清单） | 更具仪式感的「一定要完成」愿望 | 看一次极光、去一次撒哈拉、坐一次热气球 |
| 情侣旅行清单 | 适合两人世界的浪漫目的地 | 浪漫海岛、温泉小镇、适合求婚的景点与餐厅 |
| 闺蜜旅行清单 | 适合姐妹团的好拍好玩去处 | 出片胜地、精致下午茶、网红打卡集合 |
| 亲子旅行清单 | 适合带娃的安全省心去处 | 亲子乐园、自然科普、遛娃友好餐厅与酒店 |

- **F15 主题清单模板**：内置上述多种主题清单，一键创建即获得内置目的地 / 景点 / 美食推荐，开箱即用。
- **F16 清单内容收集与内置推荐**：浏览推荐或景点时一键收藏进对应清单，支持自定义新建清单；**内容库 = 第三方数据（高德）+ 用户共建**。
- **F17 完成进度与打卡联动**：清单项可标记「已完成 / 已打卡」，展示完成进度，与实际行程打卡联动（想去 → 计划中 → 已完成 → 暂时放弃）。
- **F18 清单转行程**：从清单勾选若干项，一键生成一次行程的景点池，让「愿望」自然走向「真正出发」。
- **验收标准**：可一键创建主题清单并获得内置推荐；清单项可收藏、标记完成、展示进度；可勾选若干项生成行程景点池。

### 5.6 特色智能推荐（F19–F23，V3.0）

> 把推荐从「千人一面的热门榜单」升级为**基于用户身份与情绪的个性化推荐**。**优先验证「心情推荐」维度（门槛最低、最易跑通），再扩展 MBTI，星座作趣味补充**。表达上定位为「趣味个性化偏好标签」，而非「精准算法」。

- **F19 旅行画像建立**：引导用户填写心情 / MBTI / 星座，或通过趣味问答生成「旅行人格」，低门槛建立个性化基础。优先级：**心情 → MBTI → 星座**。
- **F20 心情即时推荐**：首页提供「选择此刻心情」入口（治愈 / 解压 / 热血 / 放空 / 浪漫等），一键生成对应推荐，把模糊情绪转化为具体目的地。
- **F21 三件套推荐**：围绕画像 / 心情，同时推荐「目的地 + 景点 + 美食」成套方案，直接给出可执行的旅行雏形。推荐理由需可解释（如「因为你选了低能量 + 独自旅行，优先推荐交通方便、活动密度低的目的地」）。
- **F22 推荐转行程**：任一推荐结果可一键「加入心愿清单」或「生成行程」，无缝衔接已有规划闭环。
- **F23 推荐卡片分享**：生成个性化推荐卡片（如「INFP 的治愈之旅」）可分享，作为社交货币带来自传播。

**推荐逻辑（规则 + 内容标签，先不做复杂算法）**：每个目的地 / 景点 / 美食打上标签（独处友好、适合拍照、低体力、亲子友好、适合疗愈等）；用户输入的心情 / MBTI / 星座转化为偏好标签，再与内容标签匹配。

- **验收标准**：用户可选择心情一键生成成套推荐且理由可解释；推荐结果可一键转心愿清单 / 行程；推荐卡片可正常生成与分享。

### 5.7 独自旅行模式（F24–F28，V4.0）

> 服务快速增长的独行人群（尤其女性独行者）：缓解**安全顾虑、节奏与拍照不便、孤独感**。表达上强调「独处友好」而非「孤独叙事」，安全功能默认克制、不制造焦虑。**「搭子连接」涉及陌生人社交，不纳入本模式 MVP（见 F31）。**

- **F24 独旅模式与设置**：创建行程时选择「独自旅行模式」，自动开启安全提醒、路线强度提示、独处体验推荐与紧急联系人入口；作为一种「旅行身份」贯穿规划、记录全程。
- **F25 独行安全保障**：可设置紧急联系人、预计回程时间、夜间提醒、实时位置共享开关（**默认不公开位置，仅用户主动开启时分享**）、定时「安全打卡」提醒与一键求助入口。明确为辅助提醒，不承诺实时救援。
- **F26 独行节奏行程**：每日路线按景点数量、通勤跨度、夜间活动、步行强度给出「轻松 / 适中 / 偏累」标签，帮助避免行程过满。
- **F27 独行拍照助手**：标注景点「最佳独拍机位」、推荐可帮忙拍照的热门点位、提供自拍 / 三脚架友好提示。
- **F28 独处友好推荐与今日独处任务**：推荐一人友好的咖啡馆 / 书店 / 展览 / 一人食餐厅；每天给一个低压力任务（如「拍一张街角照片」「找一家店坐 30 分钟」），把打卡变成情绪价值记录。
- **验收标准**：开启独旅模式后规划自动偏向独行节奏、打卡提供独拍提示、手账提供个人叙事模板；位置共享默认关闭、仅主动开启时生效。

### 5.8 协作与增强（F29–F34，V5.0）

#### F29 手账多人共编

**功能描述**：手账所有者可通过"生成分享链接 + 开启协作权限 + 添加可编辑人员"邀请他人共同编辑同一篇手账。第一版（V1.0）不支持，归属 V5.0 实现，依赖云账号体系。

**协作权限设计**：

| 角色 | 权限范围 |
|-|-|
| 所有者 Owner | 编辑全部内容、删除手账、生成/关闭分享链接、邀请/移除成员、修改成员权限 |
| 编辑者 Editor | 增删改手账内容（文字/图片块）、上传素材；不可删除手账或修改权限 |
| 评论者 Commenter | 查看 + 对内容添加评论/批注；不可修改正文 |
| 查看者 Viewer | 只读 |

**操作规则**：

- 仅所有者可发起分享；协作者需登录账号后方可加入。
- 通过分享链接加入，所有者可设置链接默认权限（查看/评论/编辑）与有效期，并可随时关闭链接。
- 支持按账号/邮箱精确"添加可编辑人员"并逐人授权；可随时移除成员或调整其权限，变更后实时生效。
- 并发编辑以"内容块"为粒度，编辑时加轻量锁或以最后保存为准并提示冲突，避免内容互相覆盖；保留编辑记录（谁、何时、改了什么）。

**验收标准**：

- 所有者可生成/关闭分享链接，并设置默认权限与有效期。
- 被邀请者登录后按所授权限进入，权限边界正确（如查看者不可编辑、评论者不可改正文）。
- 多人同时编辑不丢内容、冲突有提示、编辑记录可追溯。
- 移除成员或降权后，对应权限立即失效。

#### F30 主题清单多人协作

**功能描述**：清单所有者可通过"生成分享链接 + 开启协作权限 + 添加协作人"邀请情侣/闺蜜/亲子等共同维护同一份主题清单。第一版（V1.0）不支持，归属 V5.0 实现，依赖云账号体系。

**协作权限设计**：

| 角色 | 权限范围 |
|-|-|
| 所有者 Owner | 增删清单项、勾选完成、生成/关闭分享链接、邀请/移除成员、修改权限、删除清单 |
| 编辑者 Editor | 增删清单项、勾选完成/打卡、添加推荐项；不可删除清单或改权限 |
| 评论者 Commenter | 查看 + 评论、参与"想去"投票；不可增删清单项 |
| 查看者 Viewer | 只读 |

**操作规则**：

- 仅所有者可发起分享；协作人需登录后加入。
- 分享链接可设默认权限与有效期，可随时关闭；支持按账号邀请协作人并逐人授权。
- 支持多人共同投票决定去哪；勾选/投票/增删项在多端实时同步。
- 可随时移除成员或调整权限，变更实时生效；保留操作记录（操作人与时间）。

**验收标准**：

- 协作人按所授权限维护清单，权限边界正确。
- 清单项的增删、勾选、投票在多端实时同步、不冲突。
- 移除/降权后对应权限立即失效，操作记录可追溯。
- **F31 独行搭子连接**：基于同目的地 / 同时段 / 同兴趣轻量匹配「拼饭 / 拼门票 / 结伴打卡」旅伴；**属陌生人社交，需通过合规与隐私评估后方可上线**。
- **F32 AI 辅助**：根据打卡素材自动生成手账初稿、文案润色、路线建议；需定义质量评估、失败兜底与成本上限。
- **F33 手账导出与只读分享**：导出长图 / 生成只读链接；分享前必须预览，默认隐藏精确地址、消费金额、私密备注，链接可随时关闭失效。
- **F34 地理位置智能排序**：自动计算顺路顺序、预估通勤时间，优化每日路线。
- **验收标准**：协作功能权限清晰、可控；分享默认保守、强制预览；AI 输出有兜底；智能排序结果可被用户手动覆盖。

#### F35 自定义界面风格

**功能描述**：支持用户自定义 App 界面风格主题（如极简风、莫兰迪色系、极繁主义等），可切换配色、背景与图标风格，并与手账模板体系打通；风格设置随云账号同步、多端一致。属个性化体验增强，归属 V2.0。

**素材来源方案（已调研）**：

| 来源类别 | 代表平台 | 优点 | 缺点/版权风险 |
|-|-|-|-|
| 开源组件主题库（推荐技术底座） | Arco Design Mobile、daisyUI、Material 3 | 支持 CSS 变量/Design Tokens 动态换肤，集成度最高；多为 MIT/Apache 2.0 可免费商用 | 风格偏标准化，特色视觉需自行定制 |
| 图标库 | IconPark（字节开源，Apache 2.0）、iconfont | 可免费商用，支持在线统一调整风格并生成代码 | 需规范管理图标集 |
| 可商用图库（背景/壁纸） | Unsplash API、Pexels | 海量高质量、免版税商用、无需署名 | 需走 API 接入与内容审核 |
| 配色体系 | Coolors、Adobe Color | 快速获取莫兰迪等风格科学配色，支持可访问性检查 | 仅给色值，需自建 Design Tokens 规范 |
| 设计资源社区（仅作灵感） | Figma Community、UI8、Behance | 质量高、风格前卫 | 版权风险高，禁止直接复刻 Behance/Pinterest 界面；付费资源成本高 |

**落地建议**：MVP/初期最稳妥方案 = 开源组件库主题（Arco/daisyUI，色值配置换肤）+ IconPark 图标 + Unsplash API 背景图 + 自建莫兰迪/极简/极繁 Design Tokens；严禁直接采集第三方界面图作为预设皮肤。

**版本归属**：V2.0（与主题清单、手账模板同期，复用模板/主题底座，研发效率最高）。

**验收标准**：

- 用户可在"我的/设置"中选择并实时预览、应用风格主题（至少含极简、莫兰迪、极繁三套预设）。
- 切换后全局生效，且随云账号同步、多端一致。
- 所用素材均为可商用/开源授权来源，无版权风险。

## 8. 数据结构与存储方案

### 8.1 存储方案

采用**云账号体系**：用户需注册 / 登录后使用，所有数据以「**用户账号**」为顶层归属并统一存储于云端，支持多端登录后一致同步；本地仅做缓存，用于弱网 / 断网下保持关键操作（打卡、手账编辑、清单收藏）可用，恢复网络后自动回传云端。数据默认仅本人可见。

### 8.2 核心数据对象（以账号为顶层）

```Plain Text
User（用户账号）
 ├─ Trip（行程）
 │   ├─ Spot（景点）
 │   ├─ CheckIn（打卡）
 │   └─ Journal（手账）→ Block（内容块）
 ├─ List（主题清单）→ ListItem（清单项）
 ├─ Profile（旅行画像：心情/MBTI/星座/偏好标签）
 └─ SoloSetting（独旅设置）

```

**User（用户账号）**：id、手机号 / 第三方标识、昵称、头像、created_at；为所有业务对象的顶层归属。

**Trip（行程）**

| 字段名 | 含义 | 示例 | 备注 |
|-|-|-|-|
| id | 行程唯一标识 | trip_001 | 系统生成 |
| user_id | 所属账号 | user_888 | 归属顶层 |
| title | 行程标题 | 成都 6 月之行 | 可自动生成 |
| destination | 目的地 | 成都 | 必填 |
| start_date / end_date | 起止日期 | 2026-06-10 / 06-13 | 必填 |
| status | 状态 | 即将出发 / 进行中 / 已完成 | 系统按日期与记录计算 |
| solo_mode | 是否独旅模式 | true / false | V4 起 |
| cover | 封面图 | img_url | 可选 |

**其余对象核心字段：**

- **Spot（景点）**：id、user_id、trip_id、name、city、address/坐标、tags、day_index（所属天数）、order（同天排序）、source（高德 / 自定义）。
- **CheckIn（打卡）**：id、user_id、trip_id、spot_id、photos、cost、note、checked_at、location（可选）。打卡是手账素材主要来源。
- **Journal（手账）**：id、user_id、trip_id、title、template、blocks（图文块列表）、updated_at。
- **Block（内容块）**：type（text / image）、content（文字或图片引用）、order。
- **List（主题清单）**：id、user_id、type（心愿/遗愿/情侣/闺蜜/亲子）、name、members（V5 协作）、visibility、sort_rule。
- **ListItem（清单项）**：id、list_id、地点/任务/美食、status（想去/计划中/已完成/暂时放弃）、want_level、note、completed_at、related_trip_id。
- **Profile（旅行画像）**：user_id、mood、mbti、constellation、preference_tags、relation、budget、days、departure_city。
- **SoloSetting（独旅设置）**：user_id / trip_id、emergency_contact、location_share（默认关闭）、night_reminder、route_intensity_pref。

## 9. 非功能需求

| 类别 | 要求 | 说明 |
|-|-|-|
| 账号鉴权 | 注册 / 登录后使用，登录态有安全校验与失效保护 | 避免账号被冒用；提供登录 / 退出 / 注销入口 |
| 云端数据安全 | 云端加密存储、传输链路 HTTPS 等加密协议 | 仅本人登录后可访问自己的行程、打卡与手账 |
| 跨端同步一致性 | 同一账号多端登录数据一致同步 | 避免端间数据冲突或丢失，冲突有合并 / 提示策略 |
| 登录态与隐私 | 个人信息功能首次使用进行权限与隐私说明 | 定位、相册、位置共享等敏感权限按系统规范申请并说明用途 |
| 性能 | 首页 / 行程详情常规网络下尽量 2 秒内可操作 | 地图、图片可分阶段加载；照片上传压缩、异步、不阻塞主流程 |
| 兼容性 | 优先适配主流 iOS / Android 机型与常见屏幕 | 核心按钮 / 文字对比度达可读标准，状态不仅依赖颜色表达 |
| 弱网 / 离线 | 打卡、文字、本地照片可本地暂存，恢复后同步 | 适配景区 / 境外 / 地铁等网络不稳定场景 |
| 数据保留 | 删除行程前二次确认，删除后进入短期可恢复状态 | 避免误删整段旅行记忆，恢复期限由产品策略确认 |
| 合规 | 位置共享、陌生人社交（F31）需专项合规与隐私评估 | 安全提醒明确为辅助、不承诺实时救援；具体隐私政策由法务确认 |

## 10. 埋点与数据指标

> MVP 阶段先观察核心链路是否跑通；核心口径为「完成旅行记录的旅行数」（至少 1 天行程、3 个景点、2 次打卡、1 份手账草稿）。

| 事件名 | 触发时机 | 关键属性 / 用途 |
|-|-|-|
| `trip_create_success` | 成功创建行程 | 目的地、天数、是否同行者；观察创建入口是否顺畅 |
| `spot_add_success` | 景点保存成功 | 来源（高德/自定义）、是否有地址、标签；观察候选池使用与地址缺失比例 |
| `route_generate` | 生成每日打卡路线 | 景点数、是否拖动排序；观察规划功能真实使用 |
| `map_jump` | 跳转第三方地图导航 | 目标 App（高德/百度）；观察导航跳转使用率 |
| `spot_checkin` | 打卡保存成功 | 是否定位、照片数、文字长度；衡量旅途记录完成质量 |
| `journal_generate` | 生成手账草稿 | 记录数、照片数、模板类型；观察沉淀转化 |
| `journal_save_success` | 手账保存成功 | 观察撰写链路是否顺畅 |
| `recommend_generate`（V3） | 生成个性化推荐 | 维度（心情/MBTI/星座）、转化动作；观察决策环节 |
| `list_create` / `list_item_complete`（V2） | 创建清单 / 完成清单项 | 清单类型、完成数；观察长期留存 |
| `journal_export`（V5） | 导出 / 分享手账 | 导出类型、隐私选项；观察最终内容价值认可度 |

**阶段观察口径（待内测确认目标值）**：激活（创建后 24h 内加 ≥3 景点占比）、参与（进行中行程每日 ≥1 条打卡占比）、产出（结束后 7 天内生成手账草稿占比）。

## 11. 风险与依赖

| 风险 / 依赖 | 影响 | 应对建议 |
|-|-|-|

| 范围膨胀 / 产品变重 | 偏离极简定位 | 分批迭代，推荐 / 清单必须能转化为行程 / 打卡 / 手账 |
