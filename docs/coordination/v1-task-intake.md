# Trippo V1.0 任务接入说明

## 本次判断

- 当前模式：`code`
- 是否属于 V1.0：是
- 本次应归入的 V1.0 部分：优先切入「行程列表与创建行程」，作为后续“景点搜索与心愿清单 → 每日路线与排序 → 景点打卡 → 手账草稿生成与查看”的起点

## 判断依据

- 当前仓库仍处于“编码前准备完成、准备进入编码”的状态。
- 文件系统中尚不存在正式前端工程脚手架，也没有 `package.json`、`tsconfig.json`、`vite.config.*` 等应用启动文件。
- `src/` 目前只有说明文档，没有实际业务代码；`tests/` 目前只有验收说明，没有测试脚本。
- 因此这次任务更适合进入 `code` 模式，而不是继续扩展 `design` 产物或 Demo。

## 与文件系统核对结果

- 未发现“文档说已存在、但文件系统实际不存在”的硬冲突。
- 但需要明确：当前项目仍只有文档、设计桥接资料和演示页，尚未进入正式业务代码实现阶段。
- 这意味着后续任何 V1.0 开发任务，都应优先落到 `src/`，不能把 `demos/` 当成正式实现目录。

## 当前任务影响范围

本次判断与后续首轮实现，预计会影响以下已有目录或文件：

- `src/`
- `tests/`
- `docs/coordination/status.md`
- `docs/coordination/project-maintenance.md`
- 必要时更新 `tests/acceptance-v1.md`

## 新增文件放置建议

如果接下来开始正式实现，新增文件最合理的位置如下：

- 应用入口与路由：`src/app/`
- 行程列表 / 创建行程 / 行程详情页面：`src/pages/trips/`
- 行程领域逻辑：`src/features/trip/`
- 本地 mock 数据与种子数据：`src/data/`
- 全局样式：`src/styles/`
- V1.0 冒烟检查或阶段性测试记录：`tests/`

## 本轮执行边界

- 只做 V1.0，不展开 V2.0-V5.0。
- 不提前实现发现页、主题清单、独旅模式、协作、AI 增强。
- 不把展示 Demo 误当成正式业务代码。
- 首个实现切片建议按以下顺序推进：

1. 初始化 `src/` 下的正式前端工程骨架
2. 先实现行程列表与创建行程
3. 再接入本地 mock 数据与本地持久化
4. 之后再进入景点、路线、打卡、手账链路

## 当前结论

当前最合理的下一步不是继续补展示，而是开始正式业务工程初始化，并以「行程列表与创建行程」作为 V1.0 的第一段实现切口。
