# Trippo 核心用户流

这份文档聚焦“用户怎么走完产品”，不是功能列表。

## 用户流 1：创建一次旅行

目标：用户从空状态进入，成功创建一段旅行。

```text
打开 App
  -> 首页空状态 / 行程列表
  -> 点击“创建行程”
  -> 输入目的地、日期、标题
  -> 创建成功
  -> 进入行程详情
```

关键要求：

- 创建入口必须足够明显
- 表单字段尽量少
- 创建成功后直接进入该行程，不让用户迷路

## 用户流 2：把想去的景点排成路线

目标：用户把收藏点变成某一天真正能走通的安排。

```text
进入行程详情
  -> 搜索 / 添加景点
  -> 景点进入心愿清单
  -> 选择某一天
  -> 加入当天路线
  -> 拖拽排序
  -> 保存当天路线
```

关键要求：

- 景点先进入心愿清单，再加入某一天
- 每日路线支持调整顺序
- 用户能看懂当前这一天是不是排得太满

## 用户流 3：旅途中打卡

目标：用户在到达景点时快速留下记录。

```text
打开当天路线
  -> 点击某个景点
  -> 进入打卡面板
  -> 添加照片 / 花费 / 感受
  -> 保存
  -> 该记录自动归入当前行程
```

关键要求：

- 打卡流程尽量短
- 用户可以只打卡，不必一次写很多文字
- 后续补内容应比第一次记录更方便

## 用户流 4：生成旅行手账

目标：用户不用重新整理素材，也能得到一份可以回看的手账。

```text
进入手账模块（底部入口或从行程进入）
  -> 读取已有打卡素材
  -> 自动生成草稿
  -> 用户微调图文块
  -> 保存
  -> 查看手账
```

关键要求：

- 自动生成必须足够轻
- 模板应帮助用户表达，不应带来复杂排版成本
- 手账更像整理，而不是重新创作

## 用户流 5：从“想出去”到“决定去哪”

目标：用户在还没决定目的地时，获得一组能转化为行动的旅行灵感。（后续版本 · V3.0）

```text
进入发现页
  -> 选择心情 / 偏好
  -> 获得推荐结果
  -> 选择目的地 / 景点 / 美食
  -> 加入心愿清单
  -> 创建行程
```

关键要求：

- 推荐必须可解释
- 推荐结果必须能转成清单或行程
- 不做停留在“测试结果页”的伪功能

## 用户流 6：独自旅行模式（后续版本 · V4.0）

目标：用户在独自旅行时获得安全感和节奏支持。

```text
创建行程
  -> 开启独旅模式
  -> 设置紧急联系人 / 提醒
  -> 查看独旅友好路线
  -> 旅途中记录
  -> 手账沉淀
```

关键要求：

- 默认表达要温柔，不制造焦虑
- 独旅不是社交化入口
- 节奏提醒与独拍提示应服务真实出行体验

## 用户流 7：收集心愿与清单（全局视图）

目标：用户在没有明确行程时也能持续积累“想去/想吃/想体验”的内容，并能在需要时转成具体行程。

```text
进入心愿单（底部入口）
  -> 按类型浏览（景点/美食/旅行地/清单）
  -> 添加到心愿单或清单
  -> 需要时关联到某个行程
```

关键要求：

- 心愿单是一级入口，不是行程的附属页面
- 内容应能按行程分类查看，同时允许“未关联行程”的合理归类
- 清单能力是心愿单内部的一种组织方式，而不是平级入口

## 用户流 8：在某个行程内查看心愿与手账（局部视图）

目标：用户在“当前这一趟旅行”的上下文里，只看与该行程相关的心愿和手账，不被全局内容打扰。

```text
进入行程详情
  -> 进入该行程心愿单（只看本行程）
  -> 进入该行程手账（只看本行程）
```

关键要求：

- 全局入口看全部，行程内入口看局部
- 用户要明确知道自己在“全局视角”还是“行程视角”
