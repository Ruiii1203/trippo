# Trippo 当前产品形态问题与功能路线图调整建议

这份文档基于当前 V2.0 产品形态验收反馈整理，目的不是直接改代码，而是先把 Trippo 当前的信息架构问题、功能入口问题和推荐调整方向说清楚。

它主要回答 4 个问题：

1. 当前产品形态最不合理的地方是什么
2. Trippo 更适合怎样的一级功能路线
3. 行程、心愿单、旅行手账之间应该怎样组织
4. 后续页面改版和交互优化应围绕什么结构推进

## 这轮验收反馈的核心判断

当前产品虽然已经实现了 V1.0 和部分 V2.0 能力，但从“用户打开 App 后的理解成本”来看，整体结构仍然偏向“按开发模块拼出来”，还不够像一个正常用户视角下的移动端旅行 App。

目前最核心的问题不是单个页面难看，而是：

- 一级功能入口不稳定
- 重要功能没有得到一级入口地位
- 行程、心愿单、旅行手账之间的关系表达不清楚
- 全局入口和“进入某个具体行程后的局部入口”没有形成清晰分层

所以这轮调整应该优先解决信息架构，而不是先做局部页面美化。

## 当前产品形态中最不合理的地方

### 1. 首页承担了过多“默认入口”职责

现在进入 Trippo 后，用户直接落在行程页，这本身没有问题，但页面表达方式更像“数据列表页”，而不是“整个 App 的主入口页”。

问题在于：

- 行程页承接了“默认首页”的角色
- 但心愿单和旅行手账并没有得到同等清晰的一级位置
- 结果是用户会把 Trippo 误解成一个“行程管理工具”，而不是“旅行规划 + 收集 + 沉淀”的完整产品

### 2. 底部主导航缺少产品真正的三大核心入口

根据当前体验和本轮验收反馈，Trippo 最核心的三个长期入口应当是：

- 行程
- 心愿单
- 旅行手账

但现有结构中：

- 心愿单更多被当成行程子页面或清单子能力
- 手账更多被当成行程下的后置模块
- 这与用户真实心智不一致

对用户来说，Trippo 并不只是“做行程”，而是同时满足：

- 先收集想去的
- 再组织一段行程
- 最后沉淀成回忆

如果底部导航没有把这三件事并列出来，产品的核心价值就表达不完整。

### 3. 心愿单被拆散了，没有形成一个真正独立的产品心智

当前“心愿单”在产品里很重要，但表达上仍然不够完整。它既像景点收藏，又和 V2.0 的清单体系并行存在，导致心智容易混乱。

当前不合理点包括：

- 心愿单没有独立一级入口
- “景点心愿”和“主题清单”之间关系不够直观
- 用户很难理解：哪些是某个行程里的心愿，哪些是长期收藏的旅行愿望

更合理的方式应当是：

**“心愿单”作为独立一级功能，清单体系作为心愿单内部的一部分，而不是平行于心愿单。**

### 4. 手账的重要性没有在入口层体现出来

旅行手账不是一个附属页面，它是 Trippo 和普通旅行管理工具拉开差异的重要能力。

如果手账只存在于某个行程详情里，用户会把它理解成“附带的编辑页”，而不是一个长期可回看、可整理、可沉淀的核心模块。

更合理的方式是：

- 底部导航有独立的“旅行手账”入口
- 同时在具体行程中也能进入“该行程对应的手账”

这样才符合“双入口、同内容、不同上下文”的产品结构。

### 5. 全局入口与行程内入口的关系没有被说清楚

Trippo 里的很多能力，本来就应该同时支持两种进入方式：

- 从底部导航的全局入口进入
- 从某个具体行程内进入该行程相关内容

这不是重复，而是必要的两层结构。

例如：

- 从底部进入“心愿单”，看到的是按行程分类的全部心愿与未归档心愿
- 从某个行程进入“心愿单”，看到的应该只是这个行程的相关内容

同理：

- 从底部进入“旅行手账”，看全部手账
- 从行程进入“旅行手账”，只看这个行程对应的手账

当前产品还没有把这种“全局视角 / 行程视角”的切换表达得足够清楚。

### 6. 行程首页的内容优先级需要重排

行程页是重要入口，但它不应该只是简单地把所有行程平铺出来。

更合理的排序应该是：

1. 进行中的行程
2. 计划中的行程
3. 已完成的历史行程

并且：

- 进行中的行程应当在视觉上显著高于其他状态
- 计划中与已完成的列表不应无限展开
- 计划中、历史完成分别最多展示 2 到 3 个，剩余通过“查看更多”承接

因为用户最关注的是“现在这一趟”，而不是所有历史数据。

### 7. 当前结构更像“开发视角的模块划分”，不是“用户视角的功能路线”

现在的页面拆分更接近：

- trip
- spot
- route
- checkin
- journal
- list

这对于开发没有问题，但对用户来说并不是产品结构。

用户不会想“我现在要进入 checkin 模块”，用户的心智是：

- 我要看看这次旅行
- 我要收藏想去的地方
- 我要回看我的旅行记录

所以 Trippo 接下来需要从“领域模块视角”回到“用户功能路线视角”。

## 推荐的一级功能路线

基于当前产品定位和这轮反馈，Trippo 更适合的一级导航应当收敛成 3 个：

- 行程
- 心愿单
- 手账

如果未来一定要补第四个入口，优先考虑“我的”或“设置”，但在当前阶段不应该让它抢占主产品结构。

### 为什么不是“发现 / 清单 / 我的”

因为这几个入口都不是当前阶段最能代表产品价值的主路径：

- `发现` 是未来增强入口，不是当前主心智
- `清单` 更适合作为“心愿单”内部的一个结构层，而不是单独一级导航
- `我的` 在当前阶段不是核心使用目标

Trippo 现阶段最重要的是把“旅行前收集 - 旅行中组织 - 旅行后沉淀”这条主线立住。

## 推荐的功能结构

### 1. 行程

这是用户管理“具体一趟旅行”的主入口。

包含：

- 正在进行中的行程
- 计划中的行程
- 已完成的行程
- 创建新行程
- 进入某一趟行程详情

行程详情中继续承接：

- 该行程的心愿单
- 该行程的每日路线
- 该行程的打卡记录
- 该行程的旅行手账

### 2. 心愿单

这是用户管理“我想去什么、我想体验什么”的主入口。

它应当是一个独立一级功能，而不是附属于行程或清单。

心愿单内部建议分层为：

- 景点心愿
- 美食心愿
- 旅行地心愿
- 主题清单

其中“主题清单”不是和心愿单平级，而是心愿单内部的一种组织方式。

心愿单页默认应按“所属行程”组织内容，同时保留“未关联行程”的归类。

建议结构：

- 全部
- 按行程查看
- 未关联行程
- 分类筛选（景点 / 美食 / 旅行地 / 清单）

### 3. 旅行手账

这是用户沉淀旅行记忆的主入口。

它应该在导航层就是一级功能，因为这是 Trippo 的独特价值之一。

旅行手账页应支持：

- 按行程分类查看
- 查看全部手账
- 查看未完成草稿
- 进入具体手账详情 / 编辑

同样也应支持“未关联具体行程”的合理归类，例如：

- 灵感草稿
- 待归档手账

## 双入口原则

Trippo 接下来应明确采用“双入口原则”：

### 全局入口

从底部导航进入时，看到的是某个功能的“全局视图”。

例如：

- 行程：看所有行程
- 心愿单：看所有心愿与清单
- 手账：看所有手账与草稿

### 行程内入口

从某个具体行程进入时，看到的是该功能的“当前行程视图”。

例如：

- 从某个行程进入心愿单，只看这个行程相关的心愿
- 从某个行程进入手账，只看这个行程对应的手账

这样既满足长期积累，又不破坏“以一趟旅行为主容器”的使用路径。

## 推荐的页面关系

```text
底部导航
  -> 行程
  -> 心愿单
  -> 旅行手账

行程
  -> 正在进行中的行程
  -> 计划中的行程
  -> 已完成的行程
  -> 创建行程
  -> 行程详情
      -> 该行程心愿单
      -> 每日路线
      -> 打卡记录
      -> 该行程手账

心愿单
  -> 全部心愿
  -> 按行程查看
  -> 未关联行程
  -> 分类筛选（景点 / 美食 / 旅行地 / 清单）
  -> 具体心愿项 / 清单详情
  -> 关联到某个行程

旅行手账
  -> 全部手账
  -> 按行程查看
  -> 草稿
  -> 手账详情 / 编辑
```

## 行程首页的推荐展示方式

### 结构建议

1. 页面标题与主操作
   - 标题：我的行程
   - 主操作：创建行程

2. 进行中的行程
   - 只展示最重要的当前行程
   - 视觉层级最高
   - 可展示进度、下一步动作、快捷入口

3. 计划中的行程
   - 展示 2 到 3 个
   - 剩余通过“查看更多”承接

4. 已完成的行程
   - 展示 2 到 3 个
   - 强调回看与手账沉淀，而不是和进行中抢注意力

### 行程状态建议

- 计划中
- 进行中
- 已完成

状态判断可以先基于日期规则完成，不必一开始引入更复杂的手动状态体系。

## 心愿单的推荐组织方式

心愿单不应只理解成“景点收藏夹”。

更合适的定义是：

**用户围绕旅行积累灵感、目标和想去内容的总入口。**

因此建议在视觉和结构上同时支持两种维度：

- 按内容类型看
- 按所属行程看

推荐内容类型：

- 景点心愿
- 美食心愿
- 旅行地心愿
- 主题清单

推荐归类方式：

- 某个具体行程
- 未关联行程

## 旅行手账的推荐组织方式

旅行手账不应只是某个行程详情里的末级页面，而应成为一个可以长期回看的主功能。

推荐结构：

- 全部手账
- 按行程分类
- 草稿
- 未关联内容

从行程进入时：

- 默认只看当前行程的手账
- 强化“这是这趟旅行的沉淀结果”

从全局进入时：

- 强化“这是你所有旅行记忆的总入口”

## 接下来最需要调整的不是视觉，而是结构

如果继续只修卡片、颜色、按钮，产品会变得更好看，但不会变得更合理。

当前最优先的调整方向应当是：

1. 重新定义底部导航
2. 重新定义心愿单与清单的关系
3. 重新定义旅行手账的入口层级
4. 明确“全局入口”和“行程内入口”的双入口规则
5. 重排行程首页的内容优先级

## 一句话结论

Trippo 当前最大的问题，不是功能不够多，而是**主功能路线没有被正确表达出来**。

接下来应该把产品重新收拢成：

**行程、心愿单、旅行手账**

然后再围绕这三条主线，重做页面入口、层级关系和后续 UI 改版。
