# Trippo V1.0 UI 优化范围与优先级

这份文档只服务当前已完成并通过回归验收的 V1.0 页面，用来明确：

1. 本轮 UI 优化到底改哪些页面
2. 哪些问题应该先改
3. 哪些内容虽然在产品蓝图里存在，但当前不进入本轮优化

本文档不直接定义代码实现，也不展开 V2.0 到 V5.0 的页面设计。

## 当前目标

当前 Trippo 已完成 V1.0 + V2.0 核心闭环，并通过回归验收。本轮工作不再扩展功能范围，而是围绕“更清楚、更顺手、更一致”做界面与交互打磨。

优化目标集中在三件事：

- 让用户更容易看懂当前在哪、下一步该做什么
- 让已实现页面在视觉和交互上更统一
- 修掉已经知道的非阻断性体验问题，提升可交付质量

## 本轮范围

本轮只覆盖当前已实现页面：

- `TripListPage`
- `CreateTripPage`
- `TripDetailPage`
- `SpotSearchPage`
- `WishlistPage`
- `RoutePage`
- `CheckInEditPage`
- `CheckInListPage`
- `JournalListPage`
- `JournalDetailPage`
- `ListListPage`
- `ListDetailPage`

这些页面对应 V1.0 + V2.0 的核心模块：

- 行程
- 景点搜索与心愿清单
- 每日路线
- 打卡（含花费统计）
- 手账（含模板）
- 主题清单

## 本轮不做

以下内容虽然在长期产品蓝图里存在，但不属于这次 UI 优化范围：

- `发现页` 及所有心情推荐 / MBTI / 星座相关页面
- `独旅模式` 页面体系
- 协作、分享、AI 辅助、智能排序
- 登录、账号、云同步、多端同步
- 真实地图 API、真实照片上传、真实后端
- 自定义界面风格（主题切换）

本轮也不做这些方向：

- 大范围信息架构重构
- 改产品边界
- 推翻现有路由结构
- 以“UI 优化”为名新增功能

## 优化原则

### 移动端优先

当前应用已经以移动端为主，本轮所有优化都默认优先考虑窄屏单手使用。按钮点击区、底部操作区、表单焦点、滚动体验，都应优先服务移动端。

### 先清楚，再精致

如果一个页面同时存在“视觉不够好看”和“路径不够清楚”两个问题，优先处理后者。Trippo 目前已经能跑通闭环，UI 优化首先要减少理解成本，而不是先叠加装饰。

### 全局一致

不要逐页各自优化，导致每个页面都变“更好看一点”，但整体更不统一。优先统一：

- 页面头部结构
- 卡片层级
- 主按钮与次按钮
- 表单输入框
- 状态标签
- 空状态
- 保存 / 成功 / 不可用反馈

### 不放大复杂度

V1.0 的定位仍然是轻量、克制。即使做视觉升级，也不引入重装饰、重配置、重运营式布局。

## 优先级

### P0

这些项优先处理，因为它们直接影响用户对 V1.0 是否“顺手、可交付”的判断。

1. **关键入口误导修正**
   - 心愿单中的“加入路线”入口不应继续表现为“功能开发中”。
   - 如果当前还未接通，就应改成更准确的引导方式；如果能接通，应优先打通。

2. **空状态体验统一**
   - 行程列表空状态
   - 心愿单空状态
   - 路线页空状态
   - 手账列表空状态
   - 打卡记录空状态
   
   这些页面都应明确告诉用户下一步，而不是只展示“暂无内容”。

3. **主流程按钮层级统一**
   - 创建行程
   - 添加景点
   - 加入路线
   - 去打卡
   - 生成手账
   - 保存
   
   这些按钮要在颜色、大小、圆角、阴影、反馈上保持一致，避免每页一个风格。

4. **行程详情页的信息优先级**
   - 进入详情后，用户应快速看懂：
     - 这次旅行是什么
     - 当前进度如何
     - 下一步该去心愿单、路线、打卡还是手账
   - 当前已接入多个模块入口，本轮要重点优化入口层级与区块顺序。

5. **表单与保存反馈统一**
   - 创建行程页
   - 打卡编辑页
   - 手账编辑页
   
   应统一输入框聚焦、错误提示、保存反馈和禁用态，不要每页各写一套感觉。

### P1

这些项不一定阻断使用，但会明显影响整体观感和连续体验，适合在 P0 完成后推进。

1. **卡片视觉系统统一**
   - 行程卡片
   - 景点卡片
   - 路线卡片
   - 打卡记录卡片
   - 手账卡片
   
   建议统一卡片的圆角、边框、阴影、内边距和标题层级。

2. **页面头部与分区标题统一**
   - 页面标题、返回、右侧操作、说明文本的结构应更一致
   - 各模块区块标题不要忽大忽小、忽轻忽重

3. **状态标签统一**
   - 行程状态
   - 已打卡 / 未打卡
   - 已加入心愿 / 未加入
   - 已生成 / 未生成
   
   状态表达应统一颜色语言和语义，减少用户重新学习成本。

4. **动效节奏统一**
   - 页面切换
   - 按钮点击
   - 卡片 hover / press
   - 底部面板展开
   - 搜索标签反馈
   
   当前已有体验打磨，本轮重点不是继续加更多动效，而是收敛快慢、强弱和触发时机。

5. **阅读密度优化**
   - Trippo 不应变成信息堆叠产品
   - 列表页、详情页、记录页要继续控制信息密度，让用户在移动端更容易扫读

### P2

这些项属于增强感知质量，可以做，但应排在 P0 和 P1 之后。

1. **封面图与占位视觉**
   - 行程封面占位
   - 空状态插图或柔和图形占位
   - 手账封面视觉增强

2. **微文案统一**
   - 按钮文案
   - 空状态提示
   - 轻反馈文案
   - 保存成功提示

3. **细节层级优化**
   - 间距更精确
   - 图标尺寸更统一
   - 分割线、背景层次更克制

## 页面级优化清单

### 行程列表页

目标：让用户一进来就知道当前有哪些旅行，以及最应该点哪一个。

优先项：

- 优化空状态，让首次用户明确知道“创建行程”是唯一主动作
- 强化进行中行程的视觉优先级
- 统一卡片信息层级：标题、地点、日期、状态、进度
- 让“创建行程”入口在滚动或列表较长时仍然清楚

### 创建行程页

目标：让创建动作更轻，不让用户被表单阻断。

优先项：

- 统一输入框、日期选择、错误提示样式
- 强化主按钮与禁用态
- 优化字段间距和视觉节奏，减少“像后台表单”的感觉
- 让成功创建后的反馈更自然

### 行程详情页

目标：让它成为 V1.0 的真正中枢页。

优先项：

- 明确顶部概览区的层级
- 统一四大入口卡片样式和重要程度
- 突出“今天去哪里 / 现在该做什么”
- 减少入口过多造成的平均注意力分散

### 景点搜索页

目标：让“搜索并加入心愿单”这件事更轻快。

优先项：

- 优化搜索框、热门标签、结果列表的层级关系
- 统一“加入 / 已加入”按钮状态
- 强化搜索空结果和引导语
- 控制标签、按钮和列表之间的视觉节奏

### 心愿清单页

目标：让“先收集，再编排”更清楚。

优先项：

- 解决“加入路线”入口的误导问题
- 统一景点卡片信息密度
- 强化“下一步去路线页”的引导
- 优化空状态，避免页面显得像半成品

### 每日路线页

目标：让用户更轻松地把景点排成一天能执行的路线。

优先项：

- 强化日期切换 Tab 的当前态
- 强化路线顺序可读性
- 明确“添加景点”“移除”“上下调整”的操作层级
- 优化无景点时的空状态和引导

### 打卡编辑页

目标：让用户快速完成记录，而不是被输入表单拖慢。

优先项：

- 强化“先写什么、后写什么”的视觉顺序
- 统一保存按钮、返回、编辑态反馈
- 减轻非必要字段的存在感
- 优化输入区域聚焦和占位提示

### 打卡记录页

目标：让用户回看时能快速理解已经记录了什么。

优先项：

- 按日期或景点的分组层级更清楚
- 卡片样式与其他记录型页面统一
- 空状态与返回路径明确

### 手账列表页

目标：让用户理解“这里是回忆沉淀”的入口，而不是孤立功能页。

优先项：

- 优化“生成草稿”作为主动作的表达
- 明确无打卡素材时的空状态引导
- 统一手账卡片的标题、更新时间、摘要层级

### 手账详情 / 编辑页

目标：让手账更像“轻整理”，而不是复杂编辑器。

优先项：

- 强化阅读态与编辑态的区分
- 优化标题、正文、日期分组、卡片块的层级
- 统一保存、返回、编辑切换反馈
- 保持内容优先，不让模板装饰压过素材本身

## 推荐执行顺序

如果本轮只做一小轮优化，建议顺序如下：

1. 全局视觉基线
   - 按钮
   - 输入框
   - 卡片
   - 标题层级
   - 状态标签

2. 主路径页面
   - 行程列表
   - 行程详情
   - 景点搜索
   - 心愿清单
   - 每日路线

3. 记录与沉淀页面
   - 打卡编辑
   - 打卡记录
   - 手账列表
   - 手账详情

4. 空状态与文案统一

5. 动效收敛与细节增强

## 完成标准

本轮 UI 优化完成后，至少应达到这些结果：

- 同一类控件看起来像同一个产品里的东西
- 主流程页面都能让用户一眼知道下一步
- 空状态不再显得像未完成页面
- 已知的入口误导问题得到修正
- 页面更统一，但没有引入 V2.0 以上的新功能
