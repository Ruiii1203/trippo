# Trippo UI 与交互说明

这份文档记录页面层面的设计规则，方便在 coding 过程中持续补充。

> **版本说明**：
> - V1.0 已实现：行程列表、行程详情、景点搜索、心愿清单、每日路线、打卡、手账
> - V2.0 已实现：主题清单、花费统计、手账模板
> - 以下标注「后续版本」的内容对应 V3.0+，仅供规划参考

## 全局原则

- 整体气质：极简、留白、温柔、层级清楚
- 不是信息流产品，不追求密集卡片轰炸
- 不是重工具面板，避免复杂配置先于主要体验
- 所有主流程优先考虑移动端单手使用

## 底部导航与入口层级（当前改版重点）

Trippo 当前更适合像常规 App 一样，把三大核心入口放在底部：

- 行程
- 心愿单
- 旅行手账

并采用“双入口规则”：

- 从底部入口进入：看到全局视图（按行程分类 + 未关联行程归类）
- 从某个行程进入：只看该行程相关内容（局部视图）

## 首页 / 行程列表

### 应包含的信息

- 行程名称
- 目的地
- 日期
- 行程状态：即将出发 / 进行中 / 已完成
- 已打卡进度
- 快速入口：创建行程

### 交互要求

- 空状态要明确说明下一步
- 列表卡片能快速看出“我现在最该点哪一个”
- 进行中的行程优先展示
- 行程状态建议统一为：计划中 / 进行中 / 已完成（计划中与已完成各最多展示 2–3 个，剩余用“查看更多”承接）

## 行程详情

### 应包含的子区域

- 行程概览
- 当天路线
- 心愿清单
- 打卡记录
- 手账入口

### 交互要求

- 进入后能立刻知道今天要去哪
- 心愿清单与每日路线要能快速切换
- 路线调整不应埋太深
- 行程详情应提供进入“该行程心愿单”和“该行程手账”的入口，并明确这是局部视图

## 路线页

### 页面目标

让用户把某一天的景点变成真正能执行的路线。

### 状态要求

- 无景点
- 已有景点但未排序
- 已排序
- 过满提醒

### 交互要求

- 拖拽排序要直接
- 能明显看到路线顺序
- 过满提醒是提示，不是强制阻断

## 打卡页

### 页面目标

让用户快速完成记录，而不是被复杂输入打断。

### 建议字段顺序

1. 照片
2. 感受
3. 花费
4. 时间 / 地点确认

### 交互要求

- 默认展示最少输入项
- 高级字段可以折叠
- 保存反馈要清晰

## 手账页

### 页面目标

把零散记录整理成完整回忆。

### 页面要求

- 有模板，但模板不能压过内容
- 自动带入打卡素材
- 可快速调整顺序和文字

### 交互要求

- 先生成草稿，再编辑
- 不做复杂画布式交互
- 保存和预览入口清晰

## 发现页（后续版本 · V3.0）

### 页面目标

让用户在不知道去哪时获得灵感。

### 交互要求

- 入口轻，不像问卷
- 推荐结果必须有理由
- 结果必须能转化为清单或行程

## 清单（V2.0 已实现，但结构将调整）

### 页面目标

清单是“心愿单的一种组织方式”，用于长期积累与主题化整理（人生/情侣/闺蜜/亲子等）。

当前改版重点：不再把“清单”当成底部一级入口，而是内嵌在「心愿单」入口内，让用户在同一个心智里完成“收集与整理”。

### 类型

- 心愿清单
- 人生清单
- 情侣清单
- 闺蜜清单
- 亲子清单
- 独旅清单

### 交互要求

- 清单项能清楚区分：想去 / 计划中 / 已完成
- 能从清单快速转成行程景点池

## 心愿单（当前改版重点）

心愿单是重要一级功能，需要独立入口，并支持：

- 分类展示：景点心愿 / 美食心愿 / 旅行地心愿 / 清单
- 按行程分类：某个行程 / 未关联行程
- 双入口：全局视图与行程内局部视图

## 独旅模式（后续版本 · V4.0）

### 页面目标

给独旅用户明确的安全感和节奏支持。

### 页面要求

- 有独旅设置入口
- 有紧急联系人和提醒设置
- 有路线强度提示
- 有独拍或独处友好提示

### 风格要求

- 不制造恐慌
- 不像风险警报中心
- 更像细心陪伴
