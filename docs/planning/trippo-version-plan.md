# 旅啵·Trippo 版本迭代规划

<callout emoji="⭐">
**结论先行：共需 5 个版本（V1.0 → V5.0）即可稳定实现用户目前提出的全部设想。** 五个版本遵循「**先跑通核心闭环、再丰富体验、补齐决策、做人群差异化、最后做协作与智能增强**」的分期逻辑；商业化当前暂不规划，待核心体验与规模验证后再评估。本规划与《旅啵·Trippo 产品需求文档 PRD 统一版》的 F 编号一一对应。
</callout>

## 1. 分期总思路

产品从「记录工具」升级为「懂你的旅行陪伴助手」，能力需要分批落地，避免一次性铺开导致产品变重、违背极简定位。分期遵循三条原则：**价值优先、复用已有能力、易验证**。

<grid>
<column width-ratio="0.330000">
**先立地基**
V1.0 跑通「规划—记录—沉淀」最小闭环 + 云账号体系，验证用户是否愿意持续用一个工具记录完整旅行。
</column>
<column width-ratio="0.330000">
**再补两翼**
V2.0 用主题清单拉长生命周期；V3.0 用智能推荐补齐出行前「去哪」的决策环节。
</column>
<column width-ratio="0.340000">
**最后增强**
V4.0 服务独旅人群做差异化；V5.0 做协作与智能增强，并把涉及合规的能力（搭子社交）放在最后。
</column>
</grid>

## 2. 各版本详细规划

### V1.0 · MVP（核心闭环 + 云账号 + 高德 API）

<callout emoji="💡">
**版本目标**：跑通「**规划 — 记录 — 沉淀**」最小完整闭环，验证「一次旅行 = 一个完整容器」的产品假设。
</callout>

**功能清单（对应 PRD F 编号）：**

- **F1 注册 / 登录与云账号体系**、**F2 多端云同步 + 弱网缓存**（数据底座）
- **F3 创建 / 编辑行程**、**F4 行程列表与详情**、**F5 景点打卡**、**F6 花费统计（基础）**
- **F7 景点浏览 / 搜索（高德 API + 自定义兜底）**、**F8 心愿清单**、**F9 打卡路线规划**、**F10 第三方地图导航跳转**、**F11 行程内搜索与筛选**
- **F12 手账编辑器（个人版）**、**F13 手账保存与查看**

**对应已确认决策**：决策 1（高德 API 底座）、决策 2（不做内置导航 / 跳第三方地图）、决策 6（云账号体系）、决策 7（手账第一版不支持多人共编）、决策 9（暂不商业化）。

---

### V2.0 · 体验丰富（主题清单 + 花费完善 + 手账模板）

<callout emoji="💡">
**版本目标**：丰富「收集体验」、拉长产品生命周期，让用户在「两次旅行之间」也有打开理由。
</callout>

**功能清单（对应 PRD F 编号）：**

- **F15 主题清单模板**（心愿 / 遗愿 / 情侣 / 闺蜜 / 亲子，**个人版**）
- **F16 清单内容收集与内置推荐**（高德 + 用户共建）
- **F17 完成进度与打卡联动**、**F18 清单转行程**
- **F6 花费统计完善**（按天 / 分类 / 人均 / 可视化）
- **F14 手账模板库丰富**
- **F35 自定义界面风格**（极简/莫兰迪/极繁等主题，复用手账模板/主题底座）

**对应已确认决策**：决策 3（内容库 = 第三方数据高德 + 用户共建）、决策 8（清单第一版不支持多人协作，后续支持）。

---

### V3.0 · 智能决策（特色智能推荐）

<callout emoji="💡">
**版本目标**：补齐出行前「**去哪**」的决策环节，降低「不知道去哪」的选择成本，提升从「打开 App」到「创建行程」的转化。
</callout>

**功能清单（对应 PRD F 编号）：**

- **F19 旅行画像建立**（**先心情 → 再 MBTI → 星座趣味补充**）
- **F20 心情即时推荐**、**F21 三件套推荐（目的地 + 景点 + 美食）**
- **F22 推荐转行程**（一键加入心愿清单 / 生成行程）
- **F23 推荐卡片分享**

**对应已确认决策**：决策 4（优先验证心情推荐，再 MBTI，星座作趣味补充）、决策 3（推荐内容库 = 高德 + 用户共建）。

---

### V4.0 · 人群差异化（独自旅行模式）

<callout emoji="💡">
**版本目标**：服务快速增长的独旅人群（尤其女性独行者），强化产品差异化与口碑；**不含陌生人社交**。
</callout>

**功能清单（对应 PRD F 编号）：**

- **F24 独旅模式与设置**、**F25 独行安全保障**（位置共享默认关闭、仅主动开启）
- **F26 独行节奏行程**（路线强度提示）、**F27 独行拍照助手**
- **F28 独处友好推荐与今日独处任务**

**对应已确认决策**：决策 5（不做陌生人社交，搭子连接不纳入本阶段，列入 V5 再评估）。

---

### V5.0 · 协作与增强

<callout emoji="💡">
**版本目标**：在体验成熟后做「协作」与「智能增强」，把个人产品扩展为可共享、更智能的旅行陪伴产品；涉及合规的能力放在最后并经评估后上线。
</callout>

**功能清单（对应 PRD F 编号）：**

- **F29 手账多人共编**、**F30 主题清单多人协作**：通过分享链接 + 权限（查看 / 评论 / 编辑）+ 添加协作人实现多人共编 / 协作（均复用云账号体系）
- **F31 独行搭子连接**（陌生人社交，**需合规与隐私评估**）
- **F32 AI 辅助**（手账初稿 / 文案润色 / 路线建议）
- **F33 手账导出与只读分享**（长图 / 只读链接 + 隐私预览）
- **F34 地理位置智能排序**

**对应已确认决策**：决策 7（手账后续版本支持多人共编）、决策 8（清单后续版本支持多人协作）、决策 5（搭子连接后续迭代再评估合规）。

---

### 商业化（当前暂不规划）

<callout emoji="💬">
**当前暂不规划商业化 / 付费点**（对应决策 9）。优先打磨核心体验，待核心体验成型与用户规模验证后，再评估会员模板、目的地内容包、增值服务等方向。
</callout>

## 4. 版本依赖关系说明

各版本并非完全独立，存在明确的能力依赖与递进关系，建议**严格按 V1.0 → V5.0 顺序推进**：

- **V1.0 是一切的地基**：云账号体系（F1/F2）是后续所有云端同步、协作、跨端能力的前提；行程 / 打卡 / 手账数据结构是清单联动、推荐转行程、独旅记录的承载对象。
- **V2.0 依赖 V1.0**：主题清单（F15–F18）是「心愿清单（F8）」的升级与扩展，清单转行程（F18）复用 V1 的行程与景点结构；其内置推荐内容库（高德 + 用户共建）也为 V3 推荐提供内容与标签积累。
- **V3.0 依赖 V1.0 + V2.0**：推荐转行程（F22）依赖行程结构与心愿清单 / 主题清单作为承接容器；推荐内容标签复用 V2 沉淀的内容库。
- **V4.0 依赖 V1.0**：独旅模式（F24）作为「旅行身份」贯穿行程规划、打卡与手账全流程，需基于已稳定的核心闭环；可与 V3 推荐结合（按独旅 + 心情推荐独处友好内容）。
- **V5.0 依赖前序全部版本**：多人共编 / 协作（F29/F30）依赖云账号体系（F1）与手账 / 清单结构；搭子连接（F31）依赖独旅模式（V4）且需单独合规评估；AI 辅助（F32）依赖打卡素材与手账结构；智能排序（F34）依赖景点坐标与路线数据。

<callout emoji="⭐">
**一句话总结**：**共需 5 个版本可稳定实现用户目前提出的全部设想**——V1.0 立地基、V2.0 拉长生命周期、V3.0 补决策、V4.0 做人群差异化、V5.0 做协作与智能增强；商业化待核心体验与规模验证后再评估。
</callout>
